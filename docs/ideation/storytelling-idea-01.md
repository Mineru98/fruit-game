# 스토리텔링 안 1: 무경험자의 2일 생존기

`slides/main.typ`의 재료를 가장 발표답게 살리는 방향은 기술 소개보다 `도전 서사`입니다.
다만 이 안에서도 핵심 메시지는 분명해야 합니다.
`AI를 썼다`가 아니라, `게임 개발 경험이 없는 웹 개발자가 Claude Code의 여러 테크닉을 적절히 써서 끝까지 완주했다`가 중심입니다.

## 콘셉트 제목

`게임 개발자 아닌 내가, AI와 2일 만에 과일 게임을 만든 이유`

## 청중 훅

게임 개발을 한 번도 안 해본 웹 개발자가, Claude Code에 그냥 `만들어줘`만 한 게 아니라
기획, 설계, 서브에이전트 위임, 코드 리뷰, 브라우저 검증, 자연어 반복 개선을 단계별로 써서 과일 게임을 완성했습니다.
처음엔 장난 같았는데, 버그가 터지고 구조가 무너질 때마다 `어떤 기능을 어떻게 써야 하는지`가 결과를 갈랐습니다.

## 감정선 5단계

1. 호기심
엑셀 수박 게임 영상 하나를 보고 `나도 되지 않을까?`라고 시작한다.

2. 과신
첫 프롬프트로 금방 될 것처럼 보인다.

3. 좌절
스페이스 키 버그, 충돌 버그, StrictMode 문제로 막힌다.

4. 돌파
기획 문서 작성, 서브에이전트 위임, 코드 리뷰, Playwright 검증, 멀티 AI 협업으로 구조가 잡힌다.

5. 성취
웹과 모바일에서 돌아가는 아케이드 게임이 완성되고, AI 협업의 방식이 남는다.

## 추천 발표 구조

1. 오프닝 훅
`게임 개발 경험 0인데, 과일 게임을 만들겠다고?`

2. 주인공 소개
웹 개발자, AI 도구 매니아, 하지만 게임 개발은 초보

3. 발화점
엑셀 수박 게임을 보고 시작한 이유

4. 첫 시도
`과일 게임 만들어줘` 한 줄 프롬프트와 초기 결과

5. 첫 위기
동작은 되는데 이상한 버그가 계속 생김

6. 디버깅 파트
StrictMode, 리스너 누수, 충돌 처리 문제

7. Claude Code 테크닉 파트
기획서 작성, 설계 우선, 서브에이전트 역할 분담, 리뷰 루프, Playwright 검증, Gemini 연동

8. 리디자인 파트
단순 구현에서 아케이드 감성으로 전환

9. 결과 공개
22개 커밋, 2일, 10세션, 실제 동작 데모

10. 결론
Claude Code를 잘 쓰는 사람은 단순 사용자가 아니라 개발 프로세스를 설계하는 디렉터다

## 전환 문장

- `저는 게임 개발자가 아닙니다. 그래서 이 이야기가 시작됐습니다.`
- `처음엔 되는 줄 알았습니다. 그런데 진짜 어려움은 그 다음부터였습니다.`
- `버그를 잡는 과정이야말로, AI와 협업의 진짜 실력 차이를 보여줬습니다.`
- `결국 완성도를 만든 건 한 번의 프롬프트가 아니라, Claude Code의 기능을 순서에 맞게 쓴 운영 방식이었습니다.`
- `한 번에 만드는 게 아니라, 계속 고치면서 진짜 게임이 됐습니다.`
- `결국 중요한 건 코드를 대신 써주는 AI가 아니라, 방향을 끝까지 잡는 사람이었습니다.`
- `이 프로젝트의 결과물은 게임 하나였지만, 진짜 성과는 협업 방식이었습니다.`

## 이 안의 장점

- 청중이 바로 이해할 수 있는 갈등이 있다. `무경험자 vs 게임 개발`
- 결과보다 과정이 더 재미있다. `첫 성공 -> 버그 -> 복구 -> 완성` 흐름이 분명하다.
- 기술 발표인데도 감정선이 있다. 실패와 회복이 들어가서 몰입도가 올라간다.
- `22개 커밋, 2일, 10세션` 같은 수치가 서사를 증명한다.
- `도구 나열`이 아니라 `주인공의 변화`로 보이기 때문에 기억에 남는다.
- `Claude Code를 잘 쓴 방법`이 자연스럽게 서사 안에 녹아서, 기술 발표의 목적도 같이 달성할 수 있다.

## 이 안에서 꼭 강조할 Claude Code 테크닉

- 처음부터 코드로 들어가지 않고 기획과 설계 문서를 먼저 만든 점
- 구현을 한 번에 시키지 않고 역할별 서브에이전트에 나눠 맡긴 점
- `왜 안 돼?` 같은 자연어 피드백을 반복 개선 루프로 쓴 점
- `code-reviewer`와 브라우저 검증으로 AI 결과를 그대로 믿지 않은 점
- Gemini 같은 외부 모델은 디자인처럼 강한 영역에만 제한적으로 쓴 점
