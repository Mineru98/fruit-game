# 스토리텔링 안 3: 한 줄 프롬프트에서 제품까지

이 방향은 `프롬프트 한 줄`이 어떻게 `발표할 만한 결과물`로 다듬어졌는지를 보여줍니다.
핵심은 AI가 마법처럼 만들어 준 것이 아니라, Claude Code의 여러 생산성 테크닉을 순서 있게 적용해서 제품 수준으로 끌어올렸다는 점입니다.

## 콘셉트 제목

`한 줄 프롬프트에서 제품까지: AI와 함께 다듬은 과일 게임`

핵심 키워드:
`모호함`, `반복`, `리뷰`, `리디자인`, `툴링`, `완성도`

## 청중 훅

처음엔 그냥 `과일 게임 만들어줘`였는데, 어떻게 하면 이게 발표할 만한 제품이 될까?
이 발표는 `AI가 코드를 써준 이야기`가 아니라, `모호한 요구를 Claude Code의 기획, 구현, 리뷰, 검증 루프로 제품 수준까지 끌어올린 과정`에 대한 이야기입니다.

## 감정선 5단계

1. 막연함
만들고 싶은 건 있었지만, 시작점은 너무 허술하다. `일단 돌아가면 되지 않나?`

2. 초안의 한계
동작은 하지만 어색하고, 버그가 있고, UX가 없다. `이게 정말 제품인가?`라는 불안이 생긴다.

3. 반복 개선
자연어 피드백을 주고받으며 기능, 입력, 물리, UI를 하나씩 다듬는다.
여기서 Claude Code는 마법이 아니라 작업 파트너이자 프로세스 엔진이 된다.

4. 리뷰와 재설계
코드 리뷰와 설계 재정비를 거치면서 `되는 코드`가 아니라 `유지 가능한 코드`로 바뀐다.

5. 제품화
리디자인, 툴링, 에셋 교체까지 끝나면 결과물은 데모가 아니라 완성된 제품처럼 보이기 시작한다.

## 추천 발표 구조

1. 오프닝 문제 제기
게임 개발 경험이 없는 사람이 AI와 함께 제품을 만들 수 있는가

2. 출발점 공개
정말 모호한 첫 프롬프트와 첫 결과물

3. 초안의 불편함
돌아가지만 부족한 점이 많은 상태

4. 첫 번째 개선 루프
입력, 물리, 충돌, 게임 루프를 다듬는 반복

5. 리뷰가 바꾼 것
코드 리뷰로 발견한 누수와 구조 문제

6. 리디자인 전환점
기능이 아니라 분위기를 바꾸며 제품처럼 보이게 만들기

7. 툴링의 힘
Claude Code의 기획, 서브에이전트, 리뷰, 브라우저 테스트, Gemini 연동이 역할을 나누는 장면

8. 반복으로 완성도 올리기
자연어 피드백이 어떻게 작은 수정들을 누적시키는지

9. 최종 결과물
`작동`에서 `보여줄 수 있음`으로 바뀐 지점

10. 회고와 교훈
좋은 결과는 한 번의 프롬프트가 아니라 좋은 반복 시스템에서 나온다

## 전환 문장

- `처음 프롬프트는 정답이 아니라 시작점이었습니다.`
- `AI가 코드를 써준 게 아니라, 제품이 되도록 계속 고쳐 갔습니다.`
- `작동하는 것과 완성된 것은 다릅니다.`
- `리뷰는 속도를 늦추는 과정이 아니라, 완성도를 올리는 과정이었습니다.`
- `리디자인은 예쁜 포장이 아니라, 제품의 인상을 바꾸는 전환점이었습니다.`
- `좋은 프롬프트보다 중요한 건, 좋은 피드백 루프였습니다.`
- `결국 프로젝트를 끝까지 밀어준 건 Claude Code의 개별 기능이 아니라, 그 기능들을 연결한 작업 방식이었습니다.`

## 이 안의 장점

- 기술 설명보다 `변화`를 앞세워서 청중이 쉽게 따라온다.
- `초안 -> 실패 -> 개선 -> 리뷰 -> 리디자인 -> 완성`이라는 제품화 서사가 있어서 현실적이다.
- AI를 만능 도구로 포장하지 않고, 반복과 검증이 필요하다는 점을 보여줘서 신뢰감이 있다.
- 개발자 청중에게 `툴이 좋아서가 아니라 프로세스가 좋아서 결과가 좋아졌다`는 메시지가 강하게 남는다.
- Claude Code를 `코드 생성기`가 아니라 `제품 완성 엔진`으로 보여줄 수 있다.

## 이 안에서 꼭 강조할 Claude Code 테크닉

- 애초에 모호한 요구를 기획과 설계 문서로 구체화한 점
- 구현, 리뷰, 검증을 한 번의 대화가 아니라 여러 루프로 나눈 점
- 작은 자연어 피드백을 계속 반영해 UX와 완성도를 올린 점
- 브라우저 테스트와 코드 리뷰를 통해 `돌아간다`를 `보여줄 수 있다`로 바꾼 점
- 리디자인에서도 로직과 표현을 분리해 안전하게 변경한 점
