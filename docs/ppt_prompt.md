## 문서화 끝판왕, SKILLS

1. 25년 10월 엔트로픽이 어마무시한 새로운 스펙을 공개했습니다. 이름하야 SKILLS.

2. SKILLS는 무엇이냐? 우리가 어떠한 업무를 할 때 워크플로우(정해진 업무 매뉴얼)에 따라서 일을 하게 되면 아주 효율적으로 일을 잘 할 수 있는 것처럼 어떠한 일의 순서대로 할 것인지를 나열을 하는 방식을 말합니다.

3. 가령 예를 들어보겠습니다. 우리가 어떠한 게임 개발을 한다고 할 때 이런식(docs/assets/mermaid-diagram.png)의 대략적인 플로우로 게임 개발을 하게 됩니다.

5. 이러한 내용은 게임 개발 분야 뿐만 아니라 다른 모든 산업에서 겪는 문제입니다. 심지어 저희 같은 지식 노동자 같은 경우에는 몸으로 때우는 것이 정말 쉽지 않습니다. 몸으로 때우는 것은 신체적인 문제가 없다면 사실 누구나 숙련과 관심도의 문제라고 생각하는게 제 개인 견해입니다.

6. 그런데 문제가 하나 발생했습니다. 지금의 인공지능은 글로 잘 명시만 디테일하게 되어 있다면 사람만큼이나 잘 해낼 수 있습니다. 이러한 방법이 바로 SKILLS 입니다.

## 클로드의 핵심 구성 요소

1. 클로드는 다음과 같이 구성이 되어 있습니다.
   - CLAUDE.md 메모리
   - mcp
   - Hooks
   - subagents
   - skills
   - agent teams
2. 각각의 개념에 대해서 자세히 말씀 드리기 보다는 이 요소들을 모두 이해하고 쓰는 것이 중요하다는 것을 알려드리고 싶습니다.
3. 여기서 사실 가장 핵심이 되는 구성 요소는 바로 서브 에이전트 입니다. 서브 에이전트는 어떠한 동작을 하도록 역할 지정이 되어 있는 에이전트 입니다.
4. 그리고 이 서브 에이전트의 가장 큰 핵심은 실시간으로 클로드 코드에서 만들어서 실행을 할 수 있다는 점과 병렬적으로 동시에 여러 에이전트를 실행할 수 있다라는 점입니다.
5. 이러한 구조로 인해서 순서대로 일처리를 할 필요가 없는 단계의 워크플로우 STEP에서 병렬적으로 처리를 빠르게 할 수 있습니다.
6. 아직은 실험 단계이지만 서브 에이전트에서 한단계 더 발전한 내용이 바로 에이전트 팀 입니다. [서브 에이전트와 에이전트 팀 참고 이미지](https://mintcdn.com/claude-code/nsvRFSDNfpSU5nT7/images/subagents-vs-agent-teams-dark.png?w=2500&fit=max&auto=format&n=nsvRFSDNfpSU5nT7&q=85&s=3be00c56c6a0dcccbe15640020be0128)
7. 서브 에이전트는 각자 역할을 수행을 한 다음에 오케스트레이터에게 결과를 보고를 하고 이 내용을 다 취합한 다음에 다음으로 넘어가게 되어 있고 중간 과정은 오케스트레이터는 관여하지 않는 구조로 동작을 합니다.
8. 반면 에이전트 팀은 에이전트들끼리 대화를 나누면서 피드백을 하는 식의 동작 구조를 가지고 있습니다. 이로 인해 더 많은 컨텍스트를 사용하게 되면 결과가 훨씬 더 좋아지게 됩니다. 이러한 이유는 사람들도 혼자만의 생각에 빠져서 아이데이션을 잘 하는 사람도 있겠지만 다른 사람들의 의견과 시각이 들어가게 되면 좀 더 보강이 되기 때문에 이러한 내용이 실제 인공지능 모델들도 서로 다른 모델들과 대화를 시켜서 토론의 결과를 기반으로 작업을 하게 되면 잘못 된 방향으로 가고 있는데 이를 그대로 내보내는 그런 불상사가 일어나게 되는 일을 방지할 수 있게 됩니다.

## 바이브 코딩으로 만들어 본 게임

1. 처음 이 발표 요청을 받아서 진행을 할 때 시작했던 바이브코딩만으로 게임 개발을 하는 것으로 어떤 것을 만들어볼지 게임 컨텐츠 선정을 고심을 했었습니다.
2. 다크소울 만들기, 마인크레프트 만들기 등을 시도했습니다. 그러나 처참하게 결과가 하나 같이 실패로 돌아갔습니다. (참고 영상 : docs/assets/dark-soul.mp4)
3. 바이브 코딩 신봉자였던 저는 왜 게임 개발은 바이브 코딩으로 제대로 하지 못했을까요? 뭐든 다 딸깍 하면 만들어 주는 것이 바이브 코딩 아니었던가요?
4. AI 잘 쓰기 위해서는 해당 분야에 대한 키워드를 잘 아는 것은 매우 중요합니다. 특히 저는 물리 엔진에 대한 이해도가 매우 부족했기 때문에 언제 어떤 그래픽 엔진을 쓰면 되는지에 대한 개념이 너무 없었습니다. 그리고 어떤 게임 장르 개발을 할지에 따라서도 엔진의 선택이 달라져야 하는데 이것에 대한 감이 전혀 없는 상태로 하다보니 똑같은 개발임에도 불구하고 제대로 동작하는 게임을 만들어보질 못했습니다.
5. 그러다가 유튜브에서 이런 영상을 보았습니다. [유튜브 링크](https://www.youtube.com/watch?v=Q7qpkNyWOw4)
   해당 영상을 정독을 해서 들어보니 어떻게 엑셀에서 해당 과일 게임을 만들었는지에 대해서 아주 자세하게 설명을 해주고 있는 것을 보았습니다.
   이것을 바로 본 저는 머리가 빠르게 돌더니 이 영상 스크립트를 바로 떠서 과일 게임을 만들기 위한 게임 가이드라인을 Opus로 빠르게 만들게 되었습니다.
6. 여기서부터가 진짜 바이브 코딩의 시작이었습니다. 그냥 "과일 게임 만들어줘"라고 한 줄 던진 것이 아니라, 먼저 기획 문서를 만들고 그 문서를 바탕으로 첫 구현을 진행했습니다. 처음 버전은 Canvas + Vanilla JavaScript 기반이었습니다. `Game.js`, `Fruit.js`, `Physics.js`, `CollisionDetector.js`, `GameState.js`, `InputHandler.js` 같은 식으로 역할을 나눠서 만들게 했고, 중력은 어떻게 줄지, 마찰은 어떻게 줄지, 충돌은 중심 거리와 반지름 합으로 어떻게 판단할지, 과일이 겹쳤을 때는 어떻게 분리할지까지 문서 기반으로 시켰습니다. 그 결과 처음부터 "일단 돌아가는 프로토타입"이 아니라 물리적으로 설명 가능한 게임이 나왔고, 이 단계에서만도 Jest 단위 테스트 51개를 통과시키면서 기본 동작을 잡았습니다. 제가 직접 한 줄 한 줄 구현하지 않았지만, 무엇을 검증해야 하는지와 어떤 단위로 쪼개야 하는지를 문서로 명확히 던졌기 때문에 가능한 결과였습니다.
7. 그런데 여기서 멈추지 않았습니다. 실제로 돌려보면 "된다"와 "쓸 만하다"는 완전히 다른 이야기였기 때문입니다. 그래서 곧바로 TypeScript + React + Cannon.js로 마이그레이션하는 설계 문서를 먼저 다시 만들었습니다. Vite를 쓰고, 게임 엔진 계층을 `engine`으로 분리하고, `Game.tsx`, `GameCanvas.tsx`, `GameUI.tsx`로 React 구조를 잡고, 물리 계산은 Cannon-es를 활용하도록 방향을 틀었습니다. 이 과정이 중요했던 이유는 바이브 코딩에서 AI가 가장 잘하는 것이 무에서 유를 만드는 것보다 이미 정리된 구조를 더 나은 구조로 재배치하는 일이기 때문입니다. 즉 "바로 구현"보다 "설계 후 구현"이 훨씬 잘 먹혔습니다. 그래서 실제 구현도 설계 문서를 기준으로 한 단계씩 옮겨갔고, 과일 데이터, 충돌 처리, 입력 처리, SVG 렌더링, UI를 분리하면서 점점 사람이 유지보수 가능한 코드 형태로 수렴했습니다.
8. 이후에는 정말 개발 일지 그대로, 사람이 팀으로 만드는 것 같은 작업이 이어졌습니다. 플레이 테스트를 붙여보니 스페이스 키를 누르면 페이지가 스크롤되는 문제가 있었고, React StrictMode 때문에 개발 모드에서 `useEffect`가 두 번 돌면서 입력 리스너가 사라지는 버그도 나왔습니다. 이런 것은 그냥 코드를 보는 것만으로는 놓치기 쉬운데, Playwright로 실제 브라우저를 열어보게 하니 바로 잡혔습니다. 또 "왜 시작할 때 공은 과일처럼 안 보이지?", "왜 스폰 프리뷰에 원형 테두리가 없지?", "왜 스페이스를 누르고 나면 현재 커서가 안 보이지?"처럼 사람이 제품을 만지면서 느끼는 디테일까지 계속 반영했습니다. 과일 SVG도 별도 생성 모델을 이용해 11개를 한 번에 디자인해 넣고, 모바일 터치 조작과 파비콘까지 붙였고, 나중에는 아예 팩맨 같은 오락실 감성으로 리디자인까지 갔습니다. 검정 배경, 네온 글로우, Press Start 2P 폰트, 1UP / HI SCORE / LEVEL 구조의 UI, 픽셀 아트 과일, 그리고 첫 프레임에서 과일이 안 보이는 비동기 로딩 레이스 컨디션까지 잡았습니다. 이 흐름을 보면 중요한 것은 AI가 한 번에 완벽한 정답을 내놓았다는 것이 아니라, 사람처럼 여러 차례 테스트하고 피드백을 먹이며 점점 더 완성도 높은 방향으로 밀어붙였다는 점입니다.
9. 여기서 결과가 더 드라마틱해졌던 이유가 바로 oh my claudecode 같은 harness의 존재였습니다. 제가 생각하는 harness는 단순히 "AI를 실행하는 도구"가 아닙니다. 긴 작업을 제대로 끝내기 위한 작업 운영체제에 가깝습니다. Anthropic이 long-running application development 글에서 설명한 것처럼, 긴 개발 작업에서 중요한 것은 일을 잘게 나누는 것, 중간 상태를 구조화된 산출물로 남기는 것, 그리고 작업하는 에이전트와 평가하는 에이전트를 분리하는 것입니다. 그냥 한 명의 AI에게 처음부터 끝까지 맡기면 컨텍스트가 길어질수록 방향을 잃거나, 자기 결과물을 지나치게 좋게 평가하는 문제가 생깁니다. 반면 harness는 planner가 스펙을 만들고, generator가 구현하고, evaluator가 실제로 실행해 보며 검증하는 흐름을 만들어 줍니다. 즉 "생성"과 "검증"을 분리해서 자기합리화를 막는 구조입니다.
10. 저도 이번 과일 게임 작업에서 똑같은 효과를 체감했습니다. 먼저 계획 문서를 만들고, 그 계획을 기준으로 구현을 시키고, 그다음에는 Playwright 같은 실행 계층에서 실제 동작을 확인하고, 필요하면 디버깅용 에이전트나 구현용 에이전트를 따로 태워서 수정하게 했습니다. 모바일 대응을 붙일 때는 모바일 입력 구현을 맡기는 실행 역할과 크로스 브라우저 관점에서 보는 리뷰 역할을 분리할 수 있었고, 리디자인을 할 때는 설계 역할과 구현 역할을 나눠서 진행할 수 있었습니다. 이 구조 덕분에 저는 코드 한 줄을 직접 치지 않고도 "무엇을 만들지", "어떤 기준으로 통과시킬지", "어디까지 다듬을지"를 계속 통제할 수 있었습니다. 다시 말해 바이브 코딩의 핵심은 손을 안 대는 것이 아니라, 손을 대야 하는 위치가 키보드 위에서 코드 입력이 아니라 문서, 검증, 오케스트레이션으로 이동하는 것입니다.
11. 그래서 저는 지금 바이브 코딩을 이야기할 때 "딸깍 한 번에 게임이 만들어졌다"라고 말하지 않습니다. 오히려 반대입니다. 게임 개발은 생각보다 훨씬 더 바이브 코딩의 본질을 드러냅니다. 장르 이해, 엔진 선택, 물리 개념, 테스트 방식, 리디자인 방향, 조작감 검증, 모바일 대응 같은 수많은 판단이 필요합니다. 이 판단을 문서와 스킬과 하네스로 구조화하지 않으면 결과가 금방 무너집니다. 반대로 이 구조를 잘 세워두면, 저는 다크소울이나 마인크래프트 같은 너무 큰 문제에서는 실패했지만, 과일 게임처럼 스코프를 적절하게 잡은 문제에서는 설계, 구현, 검증, 개선, 리디자인까지 매우 높은 밀도로 끝까지 밀어붙일 수 있다는 것을 확인했습니다. 결국 바이브 코딩은 운 좋은 한 방이 아니라, 잘게 쪼개고 계속 검증하는 개발 공정 그 자체라고 보는 것이 더 정확합니다.
