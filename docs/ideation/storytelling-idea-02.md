# 스토리텔링 안 2: 디버깅 스릴러

이 방향은 발표를 `개발기`가 아니라 `수사극`처럼 보이게 만듭니다.
다만 핵심 메시지는 `버그가 재밌었다`가 아니라, `Claude Code의 디버깅 테크닉과 검증 루프가 프로젝트를 끝까지 완성시켰다`여야 합니다.
청중이 좋아하는 건 완성품보다 `망가지는 순간과 해결 순간`이기 때문에, 버그를 이야기의 중심에 둡니다.

## 콘셉트 제목

`첫 프레임이 사라진 밤: AI와 함께 버그를 추적한 과일 게임 사건 파일`

## 청중 훅

처음엔 그냥 과일 게임 하나 만들려고 했습니다.
그런데 스페이스 키가 죽고, 과일이 안 사라지고, 첫 프레임은 비어 있었습니다.
그때부터 중요한 건 `AI가 대신 짠 코드`가 아니라 `Claude Code로 어떻게 재현하고, 가설 세우고, 검증하고, 수정했는가`가 됩니다.

## 감정선 5단계

1. 평온한 시작
`웹 개발자도 게임을 만들 수 있을까?`라는 가벼운 도전으로 출발

2. 첫 균열
스페이스 키 미동작, 과일 미삭제 같은 이상 징후가 연속 발생

3. 추적의 긴장감
Playwright로 재현하고, StrictMode와 비동기 타이밍을 좇으며 범위를 좁힘

4. 반전과 해결
Claude Code가 코드 전체 컨텍스트, Playwright 재현, 리뷰 흐름을 바탕으로 진짜 원인을 찾아냄

5. 해소와 통찰
문제를 고친 건 한 줄 코드가 아니라, 증거 기반 협업 방식이었다는 결론

## 추천 발표 구조

1. 오프닝 사건
`스페이스 키가 안 먹는다`

2. 오늘의 미션
과일 게임을 만들었는데 왜 자꾸 망가지는가

3. 첫 번째 현장검증
브라우저에서 버그 재현

4. 용의자 1
React StrictMode와 `useEffect` 이중 실행

5. 용의자 2
과일은 사라졌는데 물리 바디는 남아 있는 문제

6. 용의자 3
첫 프레임이 비는 SVG 비동기 레이스 컨디션

7. Claude Code 수사 도구들
`playwright-cli`, `debugger`, `systematic-debugging`, `code-reviewer`

8. AI와의 수사 방식
`왜 안 돼?` 한 문장으로 시작하지만, 실제로는 재현, 가설, 추적, 수정, 검증까지 이어지는 과정

9. 엔딩
버그를 고친 경험이 아니라, 버그를 다루는 태도를 얻었다

## 전환 문장

- `이 발표는 게임 제작기가 아니라 버그 수사기입니다.`
- `처음에는 기능이 없어서 문제였는데, 나중에는 기능이 있어서 문제였습니다.`
- `증상은 하나였지만, 원인은 세 군데에 흩어져 있었습니다.`
- `AI는 정답을 대신 쓰는 도구가 아니라, 범인을 좁히는 수사관이었습니다.`
- `프로젝트를 완성시킨 건 생성 능력이 아니라, Claude Code의 검증 능력이었습니다.`
- `마지막에 남은 건 코드가 아니라 협업 방식이었습니다.`
- `여기서부터는 플레이가 아니라 추적입니다.`

## 이 안의 장점

- 버그는 추상적인 기술 설명보다 드라마가 강하다.
- `StrictMode`, `레이스 컨디션`, `물리 바디 잔존`처럼 실제 사례가 강해서 긴장감이 유지된다.
- AI를 `개발 대체자`가 아니라 `디버깅 파트너`로 보여줘서 신뢰감이 있다.
- 기존 `slides/main.typ`의 재료를 거의 그대로 쓰면서도 훨씬 몰입감 있게 재배치할 수 있다.
- Claude Code의 강점이 `생성`보다 `재현, 추적, 검증, 수정`에 있다는 점을 가장 선명하게 보여줄 수 있다.

## 이 안에서 꼭 강조할 Claude Code 테크닉

- Playwright로 증상을 감이 아니라 재현 가능한 사건으로 바꾼 점
- `systematic-debugging`처럼 가설 기반으로 접근한 점
- `code-reviewer`를 사후 검증이 아니라 중간 품질 게이트로 쓴 점
- 하나의 버그를 UI, React lifecycle, physics layer까지 횡단해서 본 점
- 수정 후 다시 검증하는 루프를 반복해 완성도를 올린 점
