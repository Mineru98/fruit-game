# 바이브 코딩, 문서화, 그리고 Harness Engineering으로 게임을 완성한 이야기

이 문서는 `docs/ideation/*.md`에 흩어져 있던 아이데이션을 하나의 발표 스토리 라인으로 통합한 초안입니다.
핵심 메시지는 단순합니다.

`이 프로젝트는 AI가 알아서 만들어 준 결과물이 아니라, 문서화와 Claude Code의 여러 구성 요소를 적절히 써서 끝까지 완성시킨 결과물이다.`

발표 전체는 아래 흐름으로 읽히도록 설계합니다.

1. 바이브 코딩을 처음 믿게 된 계기
2. 문서화가 왜 중요한지
3. SKILLS와 Claude Code 구성 요소를 왜 이해해야 하는지
4. 왜 어떤 게임은 실패했고, 왜 과일 게임은 성공했는지
5. 실제로 어떻게 계획하고 구현하고 검증했는지
6. 이 모든 걸 가능하게 한 상위 개념이 왜 harness engineering인지

## 1. 바이브 코딩의 등장

제가 바이브 코딩이라는 개념에 본격적으로 꽂힌 계기는 2025년 2월 안드레 카파시가 올린 한 개의 트윗이었습니다.

- 인물 이미지: [안드레 카파시](https://i.namu.wiki/i/aqTQqNfkCQWNFvaznm3e538qAruufFOtsOJJ4jdvnaDSc3u69d-DaHkBocYFuXHNPvNvh9qGSPLTAbCXo6jlWyZz3uCXcyGhGvNkcmHazLWAUbkpFqBbOWsQ8U_JywCjDnBrO7apiRKo4hF1TPfAmg.webp)
- 트윗 출처: [X 링크](https://x.com/karpathy/status/1886192184808149383)
- 참고 이미지:
  - `docs/assets/vibe-coding-twitter-ko.png`
  - `docs/assets/vibe-coding-twitter-en.png`

당시 저는 이런 생각을 했습니다.

`테슬라에서 FSD를 설계하고 OpenAI를 공동 창업한 사람도 AI에게 코딩을 시켜서 작업을 한다는데, 내가 뭐라고 아직도 AI에게 코딩을 시키지 않고 직접만 하려고 하고 있지?`

이 생각 이후 저는 바이브 코딩을 단순한 유행어가 아니라, 실제 작업 방식으로 진지하게 받아들이기 시작했습니다.

그리고 2025년 3월에는 국내에서 비교적 이른 시점에 부산에서 바이브 코딩 오프라인 강의를 열었고, 온라인에서도 이 키워드를 적극적으로 알렸습니다.
그 흐름이 이어져 서울 쪽에서도 소식을 듣고, 카카오 그룹 엔지니어 대상 테크 세미나까지 진행하게 됐습니다.

즉 저는 바이브 코딩을 단순히 구경만 한 사람이 아니라, 꽤 초반부터 직접 실험하고 설명하던 사람이었습니다.

그 시기에 실제로 단 한 줄의 코드를 직접 쓰지 않고 프로젝트 하나를 끝낸 경험도 있었습니다.

- 작업 기간: 2일
- 실제 AI에게 코드 작업을 시킨 시간: 약 6시간
- 결과 영상: `docs/assets/miniflex.mp4`

당시 사용한 환경은 Cursor와 Sonnet 3.7 Max였고, MCP가 막 등장하던 시기라 외부 도구를 붙여서 화면을 만드는 작업 효율이 매우 높아지던 때였습니다.
대표적으로 Figma MCP를 활용해 Figma에 미리 만들어 둔 Netflix 화면을 거의 수정 없이 서비스 수준으로 옮겨온 경험도 있었습니다.

하지만 여기서 제가 진짜 중요하게 느낀 포인트는 `AI가 잘 그린다`, `AI가 빨리 만든다`가 아니었습니다.

그때도, 그리고 지금도 가장 중요하다고 느끼는 건 `문서화`입니다.

왜냐하면 AI는 막연한 지시보다 구조화된 문서를 훨씬 잘 따라가고,
긴 작업일수록 문서가 없으면 결국 방향을 잃기 때문입니다.

## 2. 문서화 끝판왕, SKILLS

2025년 10월, Anthropic은 아주 강력한 개념을 더 분명하게 밀어붙였습니다.
그 이름이 바로 `SKILLS`입니다.

SKILLS를 단순하게 설명하면 이렇습니다.

`어떤 일을 잘하기 위한 작업 매뉴얼을, AI가 재사용 가능한 형태로 구조화한 것`

사람도 익숙한 워크플로우를 따라가면 훨씬 안정적으로 일합니다.
기획을 먼저 하고, 그다음 설계를 하고, 구현 후 테스트를 하고, 마지막에 출시하는 식입니다.
AI도 마찬가지입니다.
순서와 기준이 적혀 있으면 훨씬 더 일관되게 움직입니다.

예를 들어 게임 개발은 대략 이런 흐름을 탑니다.

- 기획
- 프로토타입
- 개발
- 테스트
- 출시
- 운영

참고 이미지:
- `docs/assets/mermaid-diagram.png`

이런 단계마다 필요한 역할과 필요한 산출물은 사실 대부분 `지식 노동의 결과물`입니다.
문서, 설계, 코드 구조, 테스트 기준, 리뷰 기준이 있어야 다음 단계가 안정적으로 이어집니다.

문제는 인간은 한 번에 한 가지 일밖에 못 하고, 동시에 여러 전문 역할을 수행하기 어렵다는 점입니다.
게임 개발만 그런 게 아닙니다.
대부분의 산업에서 마찬가지입니다.

그런데 지금의 AI는 글로 잘 명시만 되어 있다면, 놀라울 정도로 많은 지식 노동을 잘 따라갑니다.
그래서 SKILLS는 단순 부가기능이 아니라, `문서화된 워크플로우를 AI가 실행 가능한 형태로 바꾸는 장치`라고 볼 수 있습니다.

다만 여기서 중요한 전제가 하나 있습니다.

`SKILLS를 제대로 쓰려면, Claude Code의 핵심 구성 요소를 이해해야 한다`

## 3. Claude Code의 핵심 구성 요소

제가 이번 발표에서 꼭 전달하고 싶은 건, Claude Code를 잘 쓴다는 것은 프롬프트 몇 개 잘 치는 문제가 아니라는 점입니다.

Claude Code는 대략 아래 구성 요소들로 이해할 수 있습니다.

- `CLAUDE.md` 메모리
- MCP
- Hooks
- Subagents
- Skills
- Agent Teams

각 개념을 백과사전처럼 길게 설명하는 것이 발표 목표는 아닙니다.
오히려 중요한 것은 이 요소들을 `어떻게 조합해야 긴 프로젝트를 끝까지 밀어붙일 수 있는가`입니다.

이 중에서 가장 핵심적인 구성 요소 하나를 꼽으라면 저는 `서브 에이전트`를 고르겠습니다.

서브 에이전트의 중요한 특징은 두 가지입니다.

1. Claude Code 안에서 실시간으로 역할을 만들어 실행할 수 있다.
2. 서로 독립적인 작업을 병렬로 동시에 처리할 수 있다.

이 점이 중요한 이유는, 실제 개발 워크플로우에는 순서대로 할 필요가 없는 단계가 많기 때문입니다.
설계 검토와 구현 준비, 리서치와 시각 아이데이션, 디버깅 가설 탐색처럼 병렬화 가능한 일이 많습니다.
서브 에이전트는 이걸 가능하게 해줍니다.

그리고 여기서 한 단계 더 확장된 개념이 `에이전트 팀`입니다.

참고 이미지:
- [서브 에이전트 vs 에이전트 팀](https://mintcdn.com/claude-code/nsvRFSDNfpSU5nT7/images/subagents-vs-agent-teams-dark.png?w=2500&fit=max&auto=format&n=nsvRFSDNfpSU5nT7&q=85&s=3be00c56c6a0dcccbe15640020be0128)

서브 에이전트는 각자 역할을 수행한 뒤 오케스트레이터에게 결과를 보고하는 구조에 가깝습니다.
중간 과정에 대한 상호 토론은 제한적입니다.

반면 에이전트 팀은 에이전트끼리 피드백을 주고받으며 움직일 수 있습니다.
그래서 더 많은 컨텍스트를 소비하지만, 그만큼 더 강한 검토와 보강이 가능합니다.

사람도 혼자 생각할 때보다, 다른 사람 의견이 섞일 때 더 나은 결과가 나오는 경우가 많습니다.
모델도 마찬가지입니다.
서로 다른 역할이나 모델이 관점을 교차 검증하게 만들면, 잘못된 방향으로 가고 있으면서도 자기 결과를 그대로 밀어붙이는 위험을 줄일 수 있습니다.

즉 Claude Code의 핵심은 `한 명의 똑똑한 AI`가 아니라,
`문서, 메모리, 역할 분리, 검증 루프를 가진 작업 시스템`에 있습니다.

## 4. 바이브 코딩으로 만들어 본 게임

이제 발표의 중심 사례로 넘어갑니다.

처음 발표 요청을 받았을 때 저는 `바이브 코딩만으로 게임 개발을 한다면 어떤 게임이 적절할까?`를 고민했습니다.
처음엔 다크소울 만들기, 마인크래프트 만들기 같은 큰 주제를 건드려봤습니다.

- 참고 영상: `docs/assets/dark-soul.mp4`

결과는 처참했습니다.
하나같이 실패했습니다.

여기서 중요한 질문이 생깁니다.

`바이브 코딩 신봉자였던 나는 왜 게임 개발에서는 잘 안 됐을까?`

답은 생각보다 단순했습니다.

`도메인 이해 없이, 문제를 너무 크게 잡았기 때문입니다.`

저는 물리 엔진에 대한 이해도가 낮았고,
어떤 그래픽 엔진을 언제 써야 하는지 감도 부족했습니다.
게임 장르에 따라 엔진 선택이 달라져야 한다는 것도 몸으로 이해하지 못한 상태였습니다.

즉 `뭘 만들어 달라고 말해야 하는지`를 충분히 모르는 상태에서 너무 큰 문제를 던졌기 때문에,
AI도 제대로 된 결과를 내기 어려웠던 것입니다.

그러다 유튜브에서 엑셀로 수박 게임을 만드는 영상을 보게 됐습니다.

- 유튜브 링크: https://www.youtube.com/watch?v=Q7qpkNyWOw4

이 영상은 단순히 결과만 보여주는 게 아니라, 엑셀 안에서 그 게임이 어떻게 작동하는지 아주 자세히 설명하고 있었습니다.
중력은 어떻게 표현하는지, 충돌은 어떻게 판단하는지, 물리적으로 어떤 가정을 쓰는지가 구체적으로 드러나 있었습니다.

이걸 본 순간 저는 방향이 바뀌었습니다.

`이건 그냥 게임 아이디어가 아니라, AI에게 줄 수 있는 훌륭한 게임 가이드라인이다.`

그래서 영상 스크립트를 정리하고,
그걸 바탕으로 Opus에게 과일 게임 가이드 문서를 빠르게 만들게 했습니다.

이 지점부터가 진짜 바이브 코딩의 시작이었습니다.

많은 사람이 바이브 코딩을 `그냥 만들어줘`라고 한 줄 던지는 것으로 생각하지만,
제가 이번 작업에서 실제로 했던 것은 그 반대였습니다.

먼저 기획 문서를 만들었습니다.
그 문서를 기준으로 첫 구현을 시켰습니다.
첫 버전은 Canvas + Vanilla JavaScript 기반이었습니다.

- `Game.js`
- `Fruit.js`
- `Physics.js`
- `CollisionDetector.js`
- `GameState.js`
- `InputHandler.js`

이렇게 역할을 나누고,
중력을 어떻게 줄지,
마찰을 어떻게 줄지,
충돌을 중심 거리와 반지름 합으로 어떻게 판단할지,
과일이 겹쳤을 때 어떻게 분리할지까지 문서 기반으로 명시했습니다.

그 결과 첫 결과물은 단순 프로토타입이 아니라, 물리적으로 설명 가능한 게임이었습니다.
이 단계에서만 Jest 단위 테스트 51개를 통과시키며 기본 동작을 잡았습니다.

제가 직접 코드를 치지 않았더라도,
무엇을 검증해야 하는지,
어떤 역할로 쪼개야 하는지,
어떤 기준으로 성공을 판단해야 하는지를 문서로 명확하게 던졌기 때문에 가능한 결과였습니다.

그런데 여기서 멈추지 않았습니다.

실제로 돌려보면 `된다`와 `쓸 만하다`는 완전히 다른 이야기였기 때문입니다.

그래서 바로 TypeScript + React + Cannon.js로 마이그레이션하는 설계 문서를 다시 만들었습니다.

- Vite 도입
- `engine` 계층 분리
- `Game.tsx`, `GameCanvas.tsx`, `GameUI.tsx` 구조 설계
- Cannon-es 기반 물리 방향 전환

이 과정이 중요했던 이유는,
AI가 무에서 유를 만드는 것보다 `이미 정리된 구조를 더 나은 구조로 재배치하는 일`을 훨씬 잘하기 때문입니다.

즉 바이브 코딩에서도 `바로 구현`보다 `설계 후 구현`이 훨씬 잘 먹혔습니다.

이후에는 정말 사람 팀이 작업하는 것 같은 흐름이 이어졌습니다.

플레이 테스트를 붙여보니:

- 스페이스 키를 누르면 페이지가 스크롤되는 문제
- React StrictMode 때문에 `useEffect`가 두 번 돌며 입력 리스너가 사라지는 문제
- 시작할 때 과일이 미리보기에서 과일처럼 보이지 않는 문제
- 스폰 프리뷰에 원형 테두리가 없는 문제
- 드롭 후 현재 커서가 보이지 않는 문제
- 모바일 터치 조작 문제
- 첫 프레임에서 과일이 안 보이는 비동기 로딩 레이스 컨디션

이런 문제들이 줄줄이 나왔습니다.

이걸 보면 중요한 것은 `AI가 한 번에 완벽한 정답을 내놓았는가`가 아닙니다.
오히려 진짜 중요한 것은 `사람처럼 여러 차례 테스트하고, 피드백을 먹이고, 더 나은 방향으로 계속 밀어붙일 수 있었는가`입니다.

과일 SVG는 별도 생성 모델을 통해 11개를 한 번에 디자인해서 넣었고,
모바일 터치와 파비콘도 붙였고,
나중에는 팩맨 같은 오락실 감성으로 리디자인까지 갔습니다.

- 검정 배경
- 네온 글로우
- Press Start 2P 폰트
- 1UP / HI SCORE / LEVEL 구조 UI
- 픽셀 아트 과일

즉 이 프로젝트는 `한 번에 뚝딱`이 아니라,
`문서 기반 설계 -> 구현 -> 테스트 -> 수정 -> 리디자인`의 연쇄였습니다.

## 5. 여기서부터가 진짜 핵심: Harness Engineering

이 흐름이 더 드라마틱해졌던 이유는, 단순히 Claude Code를 썼기 때문이 아니라
`Claude Code가 긴 작업을 잘 끝낼 수 있도록 운영체계를 같이 설계했기 때문`입니다.

제가 생각하는 harness는 단순히 `AI를 실행하는 도구`가 아닙니다.
긴 작업을 제대로 끝내기 위한 `작업 운영체제`에 가깝습니다.

Anthropic이 긴 개발 작업에 대해 이야기할 때 반복해서 강조하는 포인트도 비슷합니다.

- 일을 잘게 나눌 것
- 중간 상태를 구조화된 산출물로 남길 것
- 작업하는 에이전트와 평가하는 에이전트를 분리할 것

왜냐하면 한 명의 AI에게 처음부터 끝까지 맡기면,
컨텍스트가 길어질수록 방향을 잃거나,
자기 결과물을 지나치게 좋게 평가하는 문제가 생기기 때문입니다.

반면 harness는 이런 흐름을 만들어 줍니다.

- planner가 스펙을 만든다
- generator가 구현한다
- evaluator가 실제로 실행해 검증한다

즉 `생성`과 `검증`을 분리해서 자기합리화를 막는 구조입니다.

저는 이번 과일 게임 작업에서 이 효과를 매우 강하게 체감했습니다.

먼저 계획 문서를 만들고,
그 문서를 기준으로 구현을 시키고,
그다음에는 Playwright 같은 실행 계층에서 실제 동작을 확인하고,
필요하면 디버깅용 에이전트나 구현용 에이전트를 따로 태워서 수정하게 했습니다.

예를 들면:

- 모바일 대응을 붙일 때는 실행 역할과 리뷰 역할을 분리할 수 있었다.
- 리디자인을 할 때는 설계 역할과 구현 역할을 나눌 수 있었다.
- 버그를 잡을 때는 브라우저 재현과 코드 추적을 분리할 수 있었다.

이 구조 덕분에 저는 코드 한 줄을 직접 치지 않고도 다음 세 가지를 계속 통제할 수 있었습니다.

1. 무엇을 만들지
2. 어떤 기준으로 통과시킬지
3. 어디까지 다듬을지

다시 말해 바이브 코딩의 핵심은 손을 안 대는 것이 아닙니다.
손을 대야 하는 위치가 바뀌는 것입니다.

예전에는 손이 키보드 위에서 코드를 직접 쳤습니다.
지금은 손이 문서를 만들고, 검증 기준을 정하고, 작업을 오케스트레이션하는 쪽으로 이동합니다.

이게 바로 제가 이 발표에서 말하고 싶은 `harness engineering`의 실체입니다.

## 6. 그래서 제가 내린 결론

저는 지금 바이브 코딩을 `딸깍 한 번에 게임이 만들어졌다`고 설명하지 않습니다.
오히려 그 반대라고 말합니다.

게임 개발은 생각보다 훨씬 더 바이브 코딩의 본질을 드러냅니다.

- 장르 이해가 필요합니다.
- 엔진 선택이 필요합니다.
- 물리 개념이 필요합니다.
- 테스트 방식이 필요합니다.
- 리디자인 방향이 필요합니다.
- 조작감 검증이 필요합니다.
- 모바일 대응 판단이 필요합니다.

이 판단을 문서와 스킬과 하네스로 구조화하지 않으면 결과는 금방 무너집니다.

반대로 이 구조를 잘 세워두면,
다크소울이나 마인크래프트 같은 너무 큰 문제에서는 실패하더라도,
과일 게임처럼 스코프를 적절히 잡은 문제에서는 설계, 구현, 검증, 개선, 리디자인까지 매우 높은 밀도로 끝까지 밀어붙일 수 있습니다.

그래서 제가 내린 결론은 이렇습니다.

`바이브 코딩은 운 좋은 한 방이 아니라, 잘게 쪼개고 계속 검증하는 개발 공정 그 자체다.`

그리고 이 프로젝트를 가능하게 한 건 단순히 좋은 모델 하나가 아니라,
Claude Code의 여러 구성 요소와 문서화된 워크플로우를 묶어서
`계획-위임-리뷰-검증-반복`이 돌아가도록 만든 harness였습니다.

발표의 마지막 메시지는 결국 이 문장으로 수렴하면 됩니다.

`AI 시대 개발자의 경쟁력은 프롬프트를 그럴듯하게 쓰는 능력보다, 에이전트가 잘 일하도록 작업 시스템을 설계하는 능력에 더 가까워지고 있다.`

## 7. 발표용 압축 메시지

마지막 슬라이드나 발표 소개 문장으로는 아래 셋 중 하나를 쓸 수 있습니다.

### 버전 A

`게임 개발 경험이 없는 웹 개발자가, 문서화와 Claude Code의 harness를 이용해 과일 게임을 완성한 이야기`

### 버전 B

`이 발표는 AI가 게임을 만들어준 이야기가 아니라, Claude Code가 잘 일하도록 시스템을 설계한 이야기입니다.`

### 버전 C

`바이브 코딩의 본질은 손을 안 대는 것이 아니라, 손이 가야 할 위치를 코드에서 문서와 검증으로 옮기는 것입니다.`
