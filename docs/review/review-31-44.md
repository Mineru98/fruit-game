# slide-31 ~ slide-44 리뷰

실제 렌더 캡처(`docs/review/screenshots/slide-31.png` ~ `slide-44.png`)와 각 슬라이드 HTML을 함께 보고 작성했습니다.

## 슬라이드 31
### 좋은 점
- `Before → After` 대비가 직관적이라 메시지 방향은 한눈에 읽힙니다.
- 우측 구조도 카드가 본문 주장과 맞물려서 “막연한 요청”에서 “설계된 구조”로 넘어갔다는 근거를 줍니다.
- 전체 색감과 프레임 라인은 안정적이라 섹션 중반부의 톤은 잘 유지됩니다.

### 문제
- 좌측 `Before`, `After` 박스 폭이 너무 좁아서 항목이 낱말 단위로 과하게 줄바꿈됩니다. 발표 화면에서는 읽기보다 해독에 가깝습니다.
- 가운데 화살표가 충분히 크지만, 실제로는 좁은 텍스트 박스들 사이에서만 기능해서 전환의 시원함이 약합니다.
- 우측 구조도는 의미상 중요하지만 실제 렌더에서는 너무 작아 “설명 자료”보다는 “작은 썸네일”처럼 보입니다.

### 개선 제안
- 좌측 두 박스를 세로 적층으로 바꾸거나, 텍스트 박스 폭을 넓혀 한 줄 길이를 확보하세요.
- `Before`는 2개, `After`는 3개 정도의 핵심 문장만 남기고 나머지는 발표로 보완하는 편이 낫습니다.
- 우측 구조도는 과감히 확대하고, 핵심 블록 하나에만 강조 테두리나 주석을 추가해 시선 종착점을 분명히 하세요.

## 슬라이드 32
### 좋은 점
- 두 개의 카드로 정보 구조를 나눈 판단 자체는 명확합니다.
- 제목과 부제에서 “초기 버전인데도 설명 가능한 구조가 있었다”는 메시지는 잘 잡혀 있습니다.
- 파일 구조와 문서 명세를 병치한 논리는 발표 흐름상 자연스럽습니다.

### 문제
- 화면 대부분이 비어 보입니다. 카드 안의 텍스트 밀도가 너무 낮아 슬라이드 한 장의 에너지가 부족합니다.
- 좌측 카드의 파일명 리스트가 단순 나열이라 시각적 위계가 거의 없습니다.
- 우측 카드도 세 줄 불릿만 남아 있어, 왼쪽과 오른쪽이 둘 다 “문서 목록”처럼 보여 리듬이 단조롭습니다.

### 개선 제안
- 좌측은 파일 구조를 트리형 다이어그램이나 역할 그룹 3개로 재구성해 읽는 재미를 만드세요.
- 우측은 “중력 / 충돌 / 분리”를 각각 짧은 키워드 카드로 쪼개면 시각 밀도와 기억점이 좋아집니다.
- 한쪽에는 구조, 한쪽에는 예시 코드 조각이나 흐름 화살표를 넣어 양쪽 매체를 다르게 가져가세요.

- 둘 다 텍스트 카드라서 시각적 증거가 부족합니다. 중요한 전환 장면인데 인상은 약합니다.
- 우측 `발표용 메시지` 카드의 문단이 작고 길어, 실제로는 잘 안 읽힙니다.
- `Playwright`라는 키워드는 강한데 화면에서는 아이콘, 캡처, 재현 흐름 같은 구체적 장면이 없어 추상적으로 보입니다.

### 개선 제안
- 좌측은 bullet을 유지하되, 우측은 브라우저 캡처나 테스트 흐름 이미지로 바꿔주세요.
- 최소한 `재현 → 수정 → 재검증` 3단계 흐름을 도식으로 보여주면 훨씬 설득력이 높아집니다.
- 우측 문단은 한 줄 결론만 남기고 발표자가 구두로 풀어가는 편이 더 좋습니다.

## 슬라이드 44
### 좋은 점
- 모바일 대응, 브랜딩, 리디자인의 3축 분류는 확장 포인트를 설명하기 좋습니다.

- 텍스트를 더 줄이고, 각 카드에 대표 키워드 하나씩만 남기면 확장 단계의 속도가 더 잘 살아납니다.
