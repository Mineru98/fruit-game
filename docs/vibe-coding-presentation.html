SKILLS

워크플로우를 설명하는 문서가 아니라, 실행 가능한 개발 프로토콜로 만드는 단계.

Definition

SKILLS = 워크플로우를 코드로 정의하는 것

업무 매뉴얼 → AI가 순서대로 실행

무엇을 먼저 하고, 무엇을 검증하고, 어떤 산출물을 남길지를 일의 순서대로 선언하는 방식이다.

STEP 1

일의 순서를 적는다

계획 없이 “만들어줘”라고 던지지 않는다.

STEP 2

조건과 금지사항을 넣는다

제약이 많을수록 AI는 덜 흔들린다.

STEP 3

검증 루프를 연결한다

끝까지 통제 가능한 워크플로우가 된다.

Human Workflow

기획

문제 정의

시장, 장르, 재미 포인트를 명확히 언어화해야 한다.

프로토타입

빠른 실험

최소 기능으로 손맛과 방향성을 검증한다.

개발

멀티 디서플린

코드, 아트, 사운드, 레벨 디자인이 서로 발목을 잡는다.

테스트

끝없는 수정

버그, 밸런스, 성능 이슈는 마지막이 아니라 매일 생긴다.

인간의 한계: 물리적 시간 + 동시 작업 불가

지식 노동의 산출물은 병렬화가 어려웠다. 그래서 SKILLS가 들어올 자리가 생긴다.

Why Skills Matter

글로 디테일하게 명시되어 있다면, AI는 사람만큼 잘 해낸다

이것이 바로 SKILLS

  • 불분명한 요청보다 구조화된 지시를 훨씬 안정적으로 따른다.
  • 역할 분리, 검증 조건, 실패 처리까지 써주면 결과가 급격히 좋아진다.
  • 즉흥성보다 절차성이 강한 작업에서 특히 강하다.

바이브 코딩으로 만들어 본 게임

실패에서 출발해, 문서와 검증을 붙이자 실제 제품 품질로 수렴했던 과정.

Reality Check

처참한 실패: 다크소울, 마인크래프트

ATTEMPT 01

DARK SOULS

스코프가 너무 컸고, 전투·애니메이션·상태 머신에 대한 개념이 없이 “비슷하게”를 요구했다.

ATTEMPT 02

MINECRAFT

장르 특성, 엔진 구조, 월드 생성 개념을 모른 채 무작정 큰 문제를 던졌다.

바이브 코딩 신봉자였는데, 왜 게임은 안 됐을까?

Diagnosis

왜 실패했나?

ROOT CAUSE 01

도메인 키워드 부족

AI를 잘 쓰려면 해당 분야에서 어떤 키워드가 중요한지부터 알아야 한다.

ROOT CAUSE 02

물리 엔진 이해 부족

중력, 충돌, 마찰 같은 개념이 약하면 구현 지시도 약해진다.

ROOT CAUSE 03

장르별 엔진 선택 개념 부재

같은 게임 개발이라도 장르에 따라 써야 할 엔진과 구조가 다르다.

결국 실패는 AI 때문이 아니라, 내가 통제할 언어를 아직 갖고 있지 않았기 때문이다.

Critical Animation

유튜브 영상 → 기획 문서 → 첫 구현

  • 유튜브 설명을 정독해 엑셀 과일 게임의 핵심 규칙을 먼저 이해했다.
  • 그 내용을 바탕으로 게임 가이드라인 문서를 만들고, 역할 단위로 구현을 나눴다.
  • “그냥 만들어줘”가 아니라 “문서 기반으로 구현해”로 바뀌는 순간 결과가 달라졌다.
